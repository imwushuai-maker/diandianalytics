## 一、背景与目标

### 背景

AI 智能中台需要一套标准化的事件采集与分析体系，作为后续投放成本管控、素材分析等模块的数据底座。当前缺乏统一的事件定义规范和分析工具，导致各业务线数据孤岛、分析口径不一致。

### 目标

| 目标 | 衡量指标 |
|------|----------|
| 建立统一事件元数据中心 | 所有上报事件必须经过注册，未注册事件拒绝入库 |
| 支持业务方自助分析 | 分析配置操作无需工程师介入 |
| 打通投放数据闭环 | 转化事件可回传巨量，实现归因链路 |

### 用户角色

| 角色 | 使用场景 |
|------|----------|
| 数据产品经理 | 创建和维护事件、属性定义 |
| 运营 / 增长分析师 | 使用各类分析模块探索数据 |
| 开发工程师 | 查阅事件文档，按规范上报 |

## 三、P0 — 元数据管理

### 设计原则

- 属性全局注册，事件引用属性，避免重复创建
- 公共属性由系统内置，强制所有事件携带，不可删除
- 初期版本创建即上线，不设审批流，优先保障业务上线效率

### 系统内置公共属性（不可修改）

以下属性由 SDK / API 自动采集，所有事件强制携带：

| 属性名 | 字段名 | 类型 | 说明 |
|--------|--------|------|------|
| 事件时间 | `event_time` | Datetime | 事件发生的客户端时间，精确到毫秒 |
| 服务端接收时间 | `server_time` | Datetime | 服务端接收时间，用于数据质量核查 |
| 会话 ID | `session_id` | String | 用于用户路径分析的会话标识 |
| 用户 ID | `user_id` | String | 登录用户唯一标识 |
| 设备 ID | `device_id` | String | 匿名用户标识，与 user_id 做 ID-Mapping |
| 平台 | `platform` | Enum | iOS / Android / Web / 小程序 |
| App 版本 | `app_version` | String | 如 3.2.1 |
| 操作系统版本 | `os_version` | String | 如 iOS 17.2 |
| 设备型号 | `device_model` | String | 如 iPhone 15 |
| 网络类型 | `network_type` | Enum | WiFi / 4G / 5G |
| 渠道 | `channel` | String | 投放渠道标识，对接巨量归因 |
| 国家 | `country` | String | IP 解析 |
| 城市 | `city` | String | IP 解析 |

### 3.1 一般事件

#### 功能描述

提供事件的创建、编辑、查询、停用能力。所有通过 API 上报的事件必须在此注册，未注册事件将被接入层拒绝。

#### 事件列表页

**页面元素**

- 搜索框：支持事件名称、事件 ID 模糊搜索
- 筛选：按状态（草稿 / 待审批 / 已上线 / 已停用）、创建人、创建时间范围筛选
- 列表列：事件显示名、事件 ID、分类、状态、关联属性数、创建人、创建时间、操作
- 操作列：编辑、停用

**状态流转**

```
创建 → 已上线（创建即上线，无需审批）

已上线 → 停用（停用后不可再次激活，需重新创建）
```

#### 创建 / 编辑事件

**基本信息**

| 字段 | 类型 | 必填 | 规则 |
|------|------|------|------|
| 事件显示名 | String | 是 | 中文名，用于界面展示，最长 30 字符 |
| 事件 ID | String | 是 | 英文+下划线，如 `btn_click_login`，全局唯一，创建后不可修改 |
| 事件分类 | Enum | 是 | 页面浏览 / 交互行为 / 业务转化 / 自定义 |
| 负责人 | User | 是 | 选择系统用户，事件数据异常时通知负责人 |
| 描述 | Text | 否 | 事件的触发时机和业务含义，供开发参考 |
| 触发平台 | Multi-Enum | 是 | iOS / Android / Web / 小程序（多选） |
| 上报方式 | Enum | 是 | 客户端 / 服务端 |

**关联属性**

- 从全局属性库中勾选该事件需要携带的事件属性
- 已内置的公共属性默认展示但不可取消勾选
- 可标记某属性为"必传"或"选传"，影响数据质量校验规则

#### 审批流

初期版本不设审批流，事件创建后直接上线，SDK 和 API 立即开始接受该事件上报。

### 3.2 事件属性

#### 功能描述

全局属性库管理。属性在此统一注册，事件创建时从此处引用，避免重复定义。

#### 属性列表页

- 搜索：属性名、属性 ID 模糊搜索
- 筛选：属性类型、数据类型、创建时间
- 列表列：属性显示名、属性 ID、数据类型、被引用事件数、创建人、创建时间、操作
- 操作列：编辑（显示名、描述、值域约束）/ 删除（仅未被任何事件引用时可操作）

#### 创建 / 编辑属性

| 字段 | 类型 | 必填 | 规则 |
|------|------|------|------|
| 属性显示名 | String | 是 | 中文名，最长 30 字符 |
| 属性 ID | String | 是 | 英文+下划线，如 `product_id`，全局唯一，创建后不可修改 |
| 数据类型 | Enum | 是 | 字符串 / 整数 / 浮点数 / 布尔值 / 日期时间 |
| 值域约束 | 条件必填 | — | 数据类型为字符串时，可设置枚举白名单；超出白名单的值上报后标记为异常 |
| 描述 | Text | 否 | 属性含义和示例值 |

#### 约束规则

- 公共属性（系统内置）不可编辑、不可删除
- 已被事件引用的属性，数据类型不可修改（防止历史数据解析异常）
- 无任何事件引用的属性可删除

### 3.3 用户属性

#### 功能描述

定义挂载在用户身上的属性，描述用户的静态或动态特征。用户属性在所有分析模块中可作为筛选和分组维度使用。

#### 与事件属性的区别

| 维度 | 事件属性 | 用户属性 |
|------|----------|----------|
| 归属 | 某次事件 | 用户本身 |
| 更新方式 | 随事件上报 | 主动调用用户属性更新 API |
| 示例 | 支付金额、商品 ID | 会员等级、注册天数、累计消费额 |

#### 创建用户属性

| 字段 | 类型 | 必填 | 规则 |
|------|------|------|------|
| 属性显示名 | String | 是 | 中文名，最长 30 字符 |
| 属性 ID | String | 是 | 英文+下划线，全局唯一，不可修改 |
| 数据类型 | Enum | 是 | 字符串 / 整数 / 浮点数 / 布尔值 / 日期时间 / 列表 |
| 更新方式 | Enum | 是 | 覆盖（取最新值）/ 累加（数值类型专用）/ 追加（列表类型专用）|
| 描述 | Text | 否 | — |

#### 系统内置用户属性（不可修改）

| 属性名 | 字段名 | 说明 |
|--------|--------|------|
| 首次访问时间 | `first_visit_time` | 用户首次触发任意事件的时间 |
| 最近访问时间 | `last_visit_time` | 用户最近一次触发任意事件的时间 |
| 累计访问天数 | `total_active_days` | 有行为的自然日去重计数 |
| 注册时间 | `register_time` | 注册事件触发时间 |
| 首次渠道 | `first_channel` | 用户首次安装时的归因渠道 |

## 四、P0 — 自定义分析

### 功能描述

最通用的事件分析模块。用户通过选择事件、配置指标、添加筛选条件和分组维度，自由探索事件数据。核心目的是验证数据跑通、满足临时分析需求。

### 页面布局

```
┌─────────────────────────────────────────────────────┐
│  配置区（左侧 / 顶部）                                │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────────┐ │
│  │ 选择事件  │ │ 选择指标  │ │  筛选条件             │ │
│  └──────────┘ └──────────┘ └──────────────────────┘ │
│  ┌──────────────────┐ ┌──────────────────────────┐   │
│  │ 分组维度          │ │  时间范围                 │   │
│  └──────────────────┘ └──────────────────────────┘   │
│  [查询]                                               │
├─────────────────────────────────────────────────────┤
│  结果区                                               │
│  ┌──────────────────────────────────────────────┐   │
│  │  图表（折线图 / 柱状图 / 数值卡片）            │   │
│  └──────────────────────────────────────────────┘   │
│  ┌──────────────────────────────────────────────┐   │
│  │  明细表格                                     │   │
│  └──────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘
```

### 配置项详情

#### 事件选择

- 支持选择 1-5 个事件进行对比分析
- 搜索框支持事件名 / 事件 ID 模糊搜索
- 仅展示状态为"已上线"的事件

#### 指标选择

每个事件独立选择指标类型：

| 指标 | 说明 | 适用场景 |
|------|------|----------|
| 事件数 | 事件触发总次数 | 点击量、浏览量 |
| 触发用户数 | 触发该事件的去重用户数（UV） | 用户覆盖 |
| 人均次数 | 事件数 / 触发用户数 | 使用频率 |
| 触发用户占比 | 触发用户数 / 时间段内总活跃用户数 | 功能渗透率 |

#### 筛选条件

- 支持按**事件属性**或**用户属性**筛选
- 多个条件之间支持 AND / OR 逻辑
- 单个条件操作符：等于 / 不等于 / 包含 / 不包含 / 大于 / 小于 / 大于等于 / 小于等于 / 为空 / 不为空

#### 分组维度

- 支持最多 2 个分组维度
- 第 1 个维度：图表和表格均展示分组对比
- 第 2 个维度：仅在明细表格中以子行展开，不叠加到图表（避免视觉混乱）
- 每个分组默认展示 Top 10 值，支持查看全部

#### 时间范围

- 快捷选项：今天 / 昨天 / 过去 7 天 / 过去 14 天 / 过去 30 天
- 自定义日期范围
- 时间粒度：按天 / 按周 / 按月

### 结果展示

**图表**

- 多事件时默认折线图对比
- 单事件时支持切换：折线图 / 柱状图 / 数值卡片
- 图表支持导出为 PNG

**数据表格**

- 按时间 × 分组维度展开的明细数据
- 支持导出 CSV

### 分析保存

- 支持将当前配置保存为"分析快照"，命名后可在"我的分析"中快速访问
- 快照记录配置参数，不存储结果数据（查看时实时计算）

## 五、P1 — 转化分析

### 功能描述

建立多步骤转化漏斗，分析用户从起点事件到终点事件的转化路径和流失情况。核心使用场景：投放用户的注册转化、付费转化链路分析。

### 核心概念

| 概念 | 说明 |
|------|------|
| 漏斗步骤 | 2-10 个有序事件，用户需按顺序触发 |
| 转化窗口期 | 用户从第 1 步到最后 1 步必须在此时间内完成，超出则视为未转化 |
| 转化率 | 某步骤转化人数 / 第 1 步触发人数 |
| 流失率 | 1 - 相邻步骤转化率 |
| 转化唯一性 | 每个用户在同一漏斗中只计一次，取各步骤首次触发时间戳参与计算 |

### 漏斗配置

#### 步骤配置

每个步骤包含：

| 配置项 | 说明 |
|--------|------|
| 事件 | 从已上线事件中选择 |
| 属性筛选条件 | 对该步骤的事件属性添加筛选，如"支付金额 > 0" |
| 步骤名称 | 自定义展示名，默认取事件显示名 |

- 步骤数量：2-10 步
- 步骤间顺序：严格有序（用户必须按步骤顺序触发）

#### 全局配置

| 配置项 | 选项 | 默认值 |
|--------|------|--------|
| 转化窗口期 | 1小时 / 6小时 / 1天 / 3天 / 7天 / 14天 / 30天 / 自定义 | 7天 |
| 分析时间范围 | 同自定义分析 | 过去 7 天 |
| 用户属性筛选 | 对整个漏斗的用户范围进行筛选 | 不限 |
| 分组维度 | 按属性拆分对比多组漏斗 | 不分组 |

### 结果展示

#### 漏斗图

- 纵向条形漏斗，每步显示：到达人数、转化率（相对第 1 步）、步骤转化率（相对上一步）、平均转化耗时
- 有分组时，同一漏斗展示多组折叠对比

#### 步骤详情

点击任意步骤条，右侧展开面板：

| 信息 | 说明 |
|------|------|
| 转化用户 | 成功完成该步骤的用户数及属性分布 |
| 流失用户 | 未完成该步骤的用户数 |
| 平均转化耗时 | 从上一步到本步的中位数时间 |

#### 流失下钻

点击流失用户数 → 跳转至**用户路径**模块，以该批流失用户为范围，以流失步骤事件为起点，查看其后续行为路径。（P2 用户路径上线后生效）

### 漏斗保存

- 支持保存漏斗配置，命名后可在"我的漏斗"中复用
- 已保存的漏斗可添加到首页看板

## 六、P1 — 留存分析

### 功能描述

分析用户在首次触发某事件后，在后续时间段内是否再次触发目标事件，用于衡量用户粘性和功能留存效果。

### 核心概念

| 概念 | 说明 |
|------|------|
| 起始事件 | 用户进入留存分析的触发事件（如注册、首次购买） |
| 回访事件 | 衡量留存的目标事件（可与起始事件相同，如"再次打开 App"） |
| 留存率 | 在第 N 天/周/月回访的用户数 / 起始用户数 |
| 留存粒度 | 按天 / 按周 / 按月统计 |

### 留存配置

| 配置项 | 说明 |
|--------|------|
| 起始事件 | 选择 1 个事件，可添加属性筛选条件 |
| 回访事件 | 选择 1 个事件，可添加属性筛选条件，默认与起始事件相同 |
| 留存粒度 | 按天 / 按周 / 按月 |
| 分析时间范围 | 选择起始事件的发生时间范围 |
| 用户属性筛选 | 对分析的用户范围进行筛选 |
| 分组对比 | 按一个用户属性或事件属性分组，对比不同群体留存差异 |

### 结果展示

#### 留存曲线

- 折线图，X 轴为时间（第 N 天/周/月），Y 轴为留存率
- 有分组时展示多条折线对比
- 鼠标悬停显示该时间点的留存率和留存人数

#### 留存矩阵表

经典留存热力表：

```
         起始用户  第1天  第2天  第3天  第7天  第14天  第30天
2026-03-27  1200    45%    38%    32%    20%    15%     8%
2026-03-28   980    43%    36%    30%    18%    14%     -
2026-03-29  1050    46%    39%    33%    21%     -      -
```

- 行：起始日期（用户在该日触发起始事件）
- 第一列"起始用户"：该日触发起始事件的去重用户数
- 后续列"第 N 天"：在第 N 天回访的留存率
- 单元格：留存率，颜色深浅表示高低（热力图）
- 点击单元格 → 查看该批留存用户的属性分布

#### 整体留存数据卡

| 指标 | 说明 |
|------|------|
| 起始用户数 | 时间范围内触发起始事件的去重用户数 |
| 次日留存率 | Day 1 平均留存率 |
| 7日留存率 | Day 7 平均留存率 |
| 30日留存率 | Day 30 平均留存率 |

### 留存分析保存

- 支持保存分析配置，命名后可复用
- 已保存的留存分析可添加到首页看板

## 七、P2 — 分布分析

### 功能描述

分析用户在某事件上的数值型属性分布情况，或统计用户触发某事件的次数分布。回答"用户的消费金额主要集中在哪个区间"、"大多数用户每天使用几次"这类问题。

### 两种分析模式

| 模式 | 分析对象 | 典型问题 |
|------|----------|----------|
| **属性值分布** | 事件属性的数值分布 | 用户支付金额分布在哪些区间？ |
| **次数分布** | 用户触发事件的次数分布 | 用户每天打开 App 几次的居多？ |

---

### 属性值分布

#### 配置项

| 配置项 | 说明 |
|--------|------|
| 事件 | 选择 1 个事件 |
| 分析属性 | 选择该事件的一个**数值型**属性（如支付金额、视频播放时长） |
| 区间划分方式 | 自动等宽分组 / 手动自定义区间 |
| 自动分组数 | 自动等宽时可选 5 / 10 / 20 组，默认 10 组 |
| 自定义区间 | 手动输入区间边界值，如 [0,100)、[100,500)、[500,∞) |
| 筛选条件 | 按事件属性或用户属性添加筛选，多条件支持 AND / OR，缩小分析范围 |
| 时间范围 | 同自定义分析 |
| 分组对比 | 按一个用户属性或事件属性分组，对比不同群体的分布形态 |

#### 结果展示

- **柱状图**：X 轴为区间，Y 轴为落入该区间的事件次数或用户数（可切换）
- **统计摘要卡片**：最小值 / 最大值 / 中位数 / 平均值 / 总计
- **明细表格**：区间、事件次数、用户数、占比

---

### 次数分布

#### 配置项

| 配置项 | 说明 |
|--------|------|
| 事件 | 选择 1 个事件 |
| 统计粒度 | 每人在时间范围内的总触发次数 / 每人每日触发次数 |
| 次数区间 | 自动分组（如 1次、2次、3-5次、6-10次、10次以上）/ 手动自定义 |
| 筛选条件 | 按事件属性或用户属性筛选 |
| 时间范围 | 同自定义分析 |
| 分组对比 | 按一个用户属性或事件属性分组，对比不同群体的次数分布 |

#### 结果展示

- **柱状图**：X 轴为触发次数区间，Y 轴为落入该区间的用户数
- **统计摘要卡片**：人均触发次数 / 中位数 / 最大值
- **明细表格**：次数区间、用户数、占比、累计占比

---

### 分组对比

两种模式的分组对比均已在各自配置项中定义，在同一图表中对比不同分组的分布形态（如新用户 vs 老用户的付费金额分布对比）。

### 分析保存

- 支持保存分析配置，命名后可复用
- 可添加到首页看板

## 八、P2 — 用户路径

### 功能描述

以时间顺序追踪用户在 App 内的操作序列，可视化用户实际走过的行为路径。与转化分析（预设漏斗）互补：转化分析看"我定义的路径转化如何"，用户路径看"用户实际在走什么路径"。

### 两种分析方向

| 方向 | 含义 | 典型问题 |
|------|------|----------|
| **起点模式** | 从某事件出发，看用户接下来做了什么 | 用户注册后，下一步最常做什么？ |
| **终点模式** | 从某事件回溯，看用户从哪里来 | 付费用户在付费前经历了哪些步骤？ |

---

### 路径配置

| 配置项 | 说明 |
|--------|------|
| 分析方向 | 起点模式 / 终点模式 |
| 起点 / 终点事件 | 选择 1 个事件作为路径的起点或终点，可添加属性筛选条件 |
| 路径窗口期 | 用户在多长时间内的连续操作算一条路径，建议默认 30 分钟，可配置 10分钟 / 30分钟 / 1小时 / 1天 |
| 展示步数 | 路径向前/向后延伸几步，范围 2-8 步，默认 5 步 |
| 时间范围 | 同自定义分析 |
| 用户属性筛选 | 对分析的用户范围进行筛选（如只看新用户的路径） |

---

### 结果展示

#### 桑基图（主视图）

**起点模式**（从左到右）：

```
                    ┌─── 浏览商品详情 (42%) ──→ 加入购物车 (28%) ──→ ...
注册完成 ──→ 首页浏览 ─┤
                    ├─── 搜索商品 (31%)    ──→ 浏览商品详情 (25%) ──→ ...
                    └─── 结束 (27%)
```

**终点模式**（从右到左，回溯来源）：

```
... ──→ 浏览商品详情 (38%) ──→ 加入购物车 ──┐
                                           ├──→ 完成支付
... ──→ 搜索商品 (35%)    ──→ 加入购物车 ──┤
                              结束 (27%) ──┘
```

- 节点宽度代表经过该节点的用户量
- 节点显示：事件显示名、用户数、占起点/终点用户的百分比
- 默认合并同一事件的不同属性值，可点击节点展开查看属性分布
- 超过 Top 5 的分支自动合并为"其他"，可点击展开，展开后最多显示 Top 20，超出部分不再展示
- **结束节点**：用户在该步骤后 30 分钟内无后续行为、会话自然结束的用户，统一归入"结束"节点，确保各节点用户数可加和对齐

#### 节点交互

| 交互 | 效果 |
|------|------|
| 点击节点 | 右侧展开面板，显示到达该节点的用户属性分布 |
| 点击"其他"节点 | 展开显示被折叠的低频路径，最多 Top 20 |
| 右键节点 | 设为新起点（重新从该节点出发分析后续路径） |

#### 下钻至用户分群

- 点击任意节点 → 弹出命名框，输入分群名称后确认保存
- 分群持久化存储，可在所有分析模块的用户筛选条件中引用
- 分群可导出用户列表（user_id 列表，CSV 格式）

---

### 与转化分析的联动

转化分析中流失节点的"查看用户路径"入口，点击后跳转至用户路径模块，参数自动带入：
- 分析方向：起点模式
- 起点事件：流失所在步骤的事件
- 用户范围：仅该批流失用户

---

### 数据说明

- 路径分析依赖事件的 `session_id` 和 `event_time` 字段，这两个字段已在公共属性中内置，所有事件强制携带
- 服务端上报事件如无 `session_id`，系统按 `user_id + 30分钟无操作` 规则自动生成会话边界

## 九、公共规范

### 事件上报 API 规范

所有事件通过统一接口上报：

```
POST /api/v1/events
Content-Type: application/json
Authorization: Bearer {app_token}
```

**请求体（单事件）**

```json
{
  "event_id": "btn_click_login",
  "event_time": "2026-04-03T10:30:00.123+08:00",
  "user_id": "u_123456",
  "device_id": "d_abcdef",
  "session_id": "s_xyz789",
  "properties": {
    "product_id": "p_001",
    "amount": 99.00
  }
}
```

**请求体（批量事件，最多 50 条）**

```json
{
  "events": [ {...}, {...} ]
}
```

**响应**

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "accepted": 2,
    "rejected": [
      {
        "event_id": "unknown_event",
        "reason": "event_not_registered"
      }
    ]
  }
}
```

**错误码**

| code | 说明 |
|------|------|
| 0 | 成功 |
| 1001 | event_id 未注册 |
| 1002 | 必传属性缺失 |
| 1003 | 属性值不在枚举白名单 |
| 1004 | app_token 无效 |
| 1005 | 批量条数超过 50 条上限 |
| 5001 | 服务端异常 |

> 批量上报部分失败策略：成功条目正常入库，失败条目在响应 `rejected` 数组中列出，整体返回 HTTP 200。

### 数据延迟说明

| 场景 | 延迟 |
|------|------|
| 实时看板数据 | ≤ 30 秒 |
| 自定义分析 | ≤ 5 分钟 |
| 留存 / 转化分析 | ≤ 1 小时（T+0） |

## 十、非功能性需求

| 类型 | 要求 |
|------|------|

| 并发 | 接入层支持峰值 10000 QPS，可水平扩展 |
